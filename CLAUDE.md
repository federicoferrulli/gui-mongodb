# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Regola per le risposte

L'utente si chiama **Keus**: rivolgiti a lui per nome in ogni risposta.

## Comandi

```bash
npm install
npm start                  # avvia il server su http://localhost:3030 (porta: env PORT)
npm run dev                # come start, ma con riavvio automatico (node --watch)
npm run shortcut           # crea i collegamenti "CodeDB" su Desktop e menu Start (Windows, pinnabili; icona public/codedb.ico)
npm run shortcut-unix      # Linux: voce .desktop nel menu applicazioni; macOS: istruzioni per il Dock
CodeDB.cmd / ./codedb.sh   # avvio "desktop": riusa l'istanza attiva o avvia il server in background e apre il browser ("stop" per fermarlo)
node test/e2e.js           # test end-to-end MongoDB
node test/e2e-mysql.js     # test end-to-end MySQL (porta: env MYSQL_PORT, default 3306)
node test/e2e-mcp.js       # test end-to-end del gateway MCP su MongoDB (porta server: env PORT)
node test/e2e-mcp-mysql.js # test end-to-end del gateway MCP su MySQL (env MYSQL_PORT/MYSQL_PASSWORD)
npm run backup -- <comando>       # CLI di backup/restore (alias di: node backup/cli.js); comandi: backup, restore, list, verify, help
node test/e2e-backup.js           # test end-to-end della CLI di backup su MongoDB (non richiede il server)
node test/e2e-backup-mysql.js     # test end-to-end della CLI di backup su MySQL (env MYSQL_PORT/MYSQL_PASSWORD; non richiede il server)
node test/e2e-dbexport.js         # test end-to-end dell'export/import di interi database via socket (Mongo + MySQL se raggiungibile)
```

I test e2e (tranne quelli di backup) richiedono **il server già avviato su :3030** e rispettivamente un **MongoDB locale su localhost:27017** o un **MySQL locale (root, password vuota)**; creano e poi ripuliscono i database `gui_mongodb_e2e` / `gui_mysql_e2e` / `gui_mongodb_e2e_mcp` / `gui_mysql_e2e_mcp` (i test MCP salvano e poi eliminano anche le connessioni `e2e-mcp` / `e2e-mcp-mysql` in `connections.ini`). I test di backup usano un `connections.ini` temporaneo via env `CODEDB_CONNECTIONS_FILE` (override rispettato anche da `server.js`: utile per avviare un'istanza di test su un'altra porta senza passphrase reale, mai quello vero) e i database `gui_mongodb_e2e_backup` / `gui_mysql_e2e_backup` (+ suffisso `_restore`). Non ci sono lint né framework di test: i test usano semplici assert con `process.exitCode`.

## Architettura

GUI web stile DBeaver, multi-database (**MongoDB** e **MySQL** via Strategy Pattern). Niente REST: **tutta la comunicazione browser↔backend passa da Socket.IO** con callback di acknowledgment; ogni risposta è `{ ok: true, ... }` oppure `{ ok: false, error }`. Unica eccezione HTTP: l'endpoint `/mcp` per i client AI (vedi `mcp/McpGateway.js`).

- `server.js` — livello di trasporto: Express serve solo `public/` come statici; gestisce `connections.ini` e delega tutti gli eventi DB alla sessione indicata dal `tabId` nel payload. Per ogni socket il server tiene una **`Map<tabId, sessione>`** (sessione = strategia dedicata + eventuale tunnel SSH, quindi un client/pool per tab, max 8 per socket); `tabId` assente = sessione `"default"` (retro-compatibile col comportamento a connessione singola). Alla disconnessione del socket vengono chiuse tutte le sessioni. Eventi: `mongo:connect/disconnect`, `connections:test/list/get/save/delete/export/import`, `db:list/search/create/rename/drop`, `db:collections`, `db:schema`, `collection:create/rename/drop`, `column:add/alter/drop`, `index:create/drop`, `collection:find/aggregate/stats/explain`, `doc:insert/update/replace/delete`, `collection:deleteMany/export/import/ddl/watch/unwatch`.
- `mcp/McpGateway.js` — gateway **MCP (Model Context Protocol)** per client AI (Claude Code/Desktop, Cursor...), montato sull'app Express da `server.js` con `attachMcp(app, deps)`: endpoint **Streamable HTTP** `POST/GET/DELETE /mcp` (SDK `@modelcontextprotocol/sdk`, risposte JSON, successore del doppio endpoint SSE deprecato). Ogni client MCP ha una sessione (header `mcp-session-id`, un `McpServer` + transport dedicati, max 32, TTL 30 min di inattività) con una propria `Map<connection_id, sessione DB>` analoga alla mappa per-tab dei socket: `connect_database` apre una strategia via `establishConnection` (stesse connessioni salvate della UI, tunnel SSH compreso; budget `MAX_GLOBAL_SESSIONS` condiviso coi socket, max 8 connessioni per sessione MCP). Tools (Fase 1, **sola lettura**): `list_saved_connections` (mai segreti), `connect_database` (solo connessioni salvate, per nome: l'AI non vede credenziali), `disconnect_database`, `get_databases_and_collections`, `get_schema`, `execute_query` — MongoDB: `filter`/`sort`/`projection`/`skip`/`limit` (find) o `pipeline` EJSON con `$out`/`$merge` vietati; MySQL: solo `sql` con whitelist del primo token (SELECT/WITH/SHOW/DESCRIBE/EXPLAIN/TABLE/VALUES), blocco `INTO OUTFILE/DUMPFILE` ed esecuzione in transazione `READ ONLY` (flag `readOnly` di `MySqlStrategy.collectionAggregate`, timeout 30 s). Strumenti analitici: `get_shortest_path` (BFS), `analyze_dependencies` (seeding order), `analyze_pii` (GDPR/PII), `audit_schema` (diagnostica salute 0-100), `filter_empty_tables` (vuote/popolate), `get_graph` (nodi ed archi JSON per rendering 3D/2D). Fase 2: risorsa `schema://{connectionId}/{db}` (Mermaid UML + dizionario dati) e `graph://{connectionId}/{db}` (struttura nodi/archi JSON), e prompts `genera-report`, `esplora-database`, `analizza-gdpr` e `diagnostica-schema`. Fase 3: tool separato `execute_write` (richiede `readOnly=false` ed anteprima/conferma con `confirm_token` monouso) e `set_connection_read_only` (doppia conferma). Backup via MCP: `backup_database`, `list_backups`, `restore_backup`.
- `backup/` — **CLI di backup e ripristino** (`backup/cli.js`, alias `npm run backup`), indipendente dal server: riusa le strategie di `db/` (tunnel SSH compreso via `backup/lib/connect.js`, clone di `establishConnection`) e legge `connections.ini` in **sola lettura** (`backup/lib/connstore.js`: mai riscritto, passphrase da `GUI_MONGO_PASSPHRASE` o prompt mascherato, chiesta solo se ci sono segreti `ENC:`; override del percorso con env `CODEDB_CONNECTIONS_FILE` per i test). Layout: `<dest>/<conn>_<db>/<id>/` (id = `timestamp_tipo`) con `manifest.json` (metadati + checksum SHA-256), `data/*.ndjson[.gz]` (una riga EJSON relaxed per documento/riga, dump in streaming: cursor MongoDB / stream mysql2, gzip `zlib`), `indexes/*.json` (Mongo, solo full), `schema/*.sql` (`SHOW CREATE TABLE`, MySQL); catalogo di gruppo in `catalog.json`. Tipi: **full**, **incremental** (modifiche dall'ultimo backup) e **differential** (dall'ultimo full), individuati da un campo data (`--since-field`; default: Mongo timestamp degli ObjectId — solo nuovi inserimenti —, MySQL auto-rilevamento di `updated_at`/`created_at` e simili con confronto `FROM_UNIXTIME` per essere indipendente dai fusi, altrimenti tabella per intero); le **cancellazioni non vengono catturate**. `restore` risolve da solo la catena risalendo i `baseId` (layer 1 = INSERT, successivi = upsert: `replaceOne` upsert / `REPLACE INTO`), supporta `--target-db`, `--collections` (selettivo) e `--drop`; `verify` ricontrolla i checksum; `list` legge i cataloghi. Storage cloud opzionale post-dump (`--storage s3://…|gs://…|azure://…`, `backup/lib/storage.js`): SDK caricati on-demand e **non** in `package.json` (l'errore spiega cosa installare), credenziali solo dai canali standard dei provider. Log operazioni (inizio/fine/stato/durata/errori) in `<dest>/backup.log` (`backup/lib/logger.js`), notifica Slack opzionale a fine operazione (`--slack-webhook` o env `SLACK_WEBHOOK_URL`, `backup/lib/notify.js`). La cartella `backups/` è gitignored.
- `db/` — Strategy Pattern: `DbStrategy.js` (interfaccia base + euristiche `detectRelations` condivise per l'UML), `MongoDbStrategy.js` (un `MongoClient` per istanza; rename = copia via `$out` cross-db + drop, MongoDB ≥ 4.4; db di sistema protetti; change stream per `collection:watch`, su standalone degrada emettendo `watch:unavailable`), `MySqlStrategy.js` (pool `mysql2`; `_id` virtuale = chiave primaria come oggetto `{ colonna: valore }`, fallback all'intera riga senza PK; `filter` = clausola WHERE libera, `sort` = SQL libero o JSON `{"col":1}`, `collection:aggregate` = SQL Raw (con `payload.readOnly`, usato dal gateway MCP, esegue in transazione `READ ONLY` con timeout 30 s); schema/relazioni da `information_schema` con FK reali + euristiche; schemi di sistema protetti; niente watch), `DbFactory.js` (`getStrategy(dbType)`).
- `public/js/` — frontend modulare (vanilla JS, nessuna build, nessun framework). Lo stato vive **per tab** in `tabs.js` (registro dei tab: ognuno ha `id` = tabId della sessione server, metadati e un proprio `state` creato da `freshState()`); `state.js` esporta un **Proxy che delega allo stato del tab attivo**, così i moduli storici leggono/scrivono "un singolo oggetto" senza sapere dei tab. Tutte le richieste passano da `emit()`/`notify()` in `utils.js`, che iniettano il `tabId` del tab attivo catturato al momento della chiamata (la risposta porta il tab d'origine in `_tab`; se il tab è stato chiuso la risposta viene scartata). Utility condivise in `utils.js`. L'entrypoint è `main.js`, che coordina i moduli:
  - `tabs.js`: Registro dei tab di connessione (createTab/switchTab/closeTab/closeAllTabs, onTabChange)
  - `tabbar.js`: Barra dei tab di connessione (stile VS Code, click centrale = chiudi, ＋ = nuova connessione)
  - `colltabs.js`: Tab di secondo livello per le collection/tabelle aperte dentro un tab di connessione (`state.collTabs`); ogni coll-tab ha uno **snapshot** di query/risultati/vista ripristinato alla riattivazione; drop/rename di db e collection chiudono/aggiornano i coll-tab (`closeCollTabsWhere`/`updateCollTabs`)
  - `connmanager.js`: Sidebar sinistra con le connessioni salvate raggruppate per cartella (`folder`), menu contestuale (apri in nuovo tab / testa / modifica / elimina)
  - `connection.js`: Modale di connessione (form) e `connectAndOpenTab` (il tab compare solo a connessione riuscita)
  - `workspace.js`: Ri-render del workspace condiviso dallo stato del tab attivo (`renderWorkspace`/`saveWorkspaceInputs`) — il DOM del workspace è **unico**, al cambio tab viene ri-popolato
  - `utils.js`: Helper vari (EJSON, stringhe, modali, socket, errori) + `emit`/`notify` col tabId
  - `dbtree.js`: Render della sidebar interna coi DB del tab attivo
  - `schema-ops.js`: Modali di creazione/alter (DDL)
  - `grid.js`: Griglia dati, query SQL/Mongo e impaginazione (`selectCollection` apre un coll-tab)
  - `inlineEdit.js`: Editing al volo della riga
  - `insert.js`: Inserimento nuovo documento/riga
  - `details.js`: Vista statistiche, indici e definizioni colonne
  - `live.js`: Logica di polling / socket watch (eventi push instradati per tabId)
  - `autocomplete.js`: Autocomplete contestuale per filtri e ordinamenti
  - `exportimport.js`: Modali di export (EJSON/CSV/SQL INSERT) e import batch con progresso e report errori; export/import di **interi database** in un file `.codedb.json` auto-contenuto (`{ formato: "codedb-database", dbType, db, collections: [{ name, ddl, indexes, docs }] }`, docs in EJSON relaxed): l'export orchestra `collection:export` format `json` (ora anche MySQL) + `collection:ddl` (MySQL) + indici da `collection:stats` (Mongo); l'import (`dbimport-overlay`, con db di destinazione e checkbox drop&ricrea) ricrea schema/indici e spedisce i batch con `collection:import`. Voci nel menu contestuale dei database e della sidebar (`dbtree.js`)
  - `queryhistory.js`: Cronologia delle query eseguite (persistita in localStorage per collection)
  - `cellselect.js`: Selezione celle stile Excel, copia multi-formato (TSV/JSON/CSV/Markdown/SQL), incolla da Excel, selezione colonne ed export CSV
  - `responsive.js`: Adattamento layout per viewport piccole (drawer ≤900px, touch/orientamento)
  - `socket.js` e `uml.js`. Il workspace ha tre viste (`setView`): **Dati** (griglia), **Dettagli** (statistiche/indici/schema) e **UML** (SVG generato in `uml.js`); la vista è ricordata per coll-tab. Menu contestuale generico (tasto destro su connessioni, database e collection). Layout con sidebar ridimensionabili (`.resizer`, larghezze in localStorage).
- `public/index.html` + `public/css/style.css` — markup e stile; `public/codedb.ico`/`codedb.png` sono favicon e icone dei collegamenti desktop (rigenerabili con `node tools/genera-icona.js`, zero dipendenze).
- `CodeDB.cmd` / `codedb.sh` + `tools/crea-collegamento.ps1`/`.sh` + `tools/avvio-nascosto.ps1` — avvio "desktop" multipiattaforma: il launcher riusa l'istanza già in ascolto (apre solo il browser) oppure chiede la passphrase con un prompt mascherato in console (Windows: `Read-Host -AsSecureString` in `avvio-nascosto.ps1`; UNIX: `read -s`; saltata se `GUI_MONGO_PASSPHRASE` è impostata) e avvia `node server.js` **in background** (Windows: `Start-Process -WindowStyle Hidden`; UNIX: `nohup`+`disown`; log in `codedb.log`/`codedb.err.log`, gitignored): console/terminale chiudibili subito; browser aperto quando la porta risponde (check: PowerShell su Windows, curl con fallback `/dev/tcp` su UNIX); argomento `stop` per fermare il server (kill del processo in ascolto sulla porta). Gli script di collegamento (`npm run shortcut` / `shortcut-unix`) creano: su Windows i .lnk su Desktop e menu Start con target `cmd.exe /c` (requisito per il pin alla barra delle applicazioni), su Linux la voce `.desktop` nel menu applicazioni (`Terminal=true` per la passphrase), su macOS stampano le istruzioni per il Dock. `server.js` gestisce `EADDRINUSE` con un messaggio esplicativo e **rifiuta di partire con passphrase errata** (`decryptFailures` in `decryptSecret`: exit 1 prima della migrazione, così i segreti non vengono mai azzerati).
- `db/SshTunnel.js` — `openSshTunnel(ssh, target)`: tunnel SSH condiviso (ortogonale al dbType, basato su `ssh2`). Apre una porta locale effimera su `127.0.0.1` che inoltra verso `target.host:target.port` sul lato remoto; auth via password **oppure** chiave privata (`sshKeyFile`, percorso su disco) + `sshPassphrase`. `server.js` apre the tunnel in `mongo:connect` quando `cfg.ssh === 'true'` (solo modalità "Parametri", non con URI completa), riscrive `host/port` verso il capo locale prima di `strategy.connect` (per MongoDB aggiunge `directConnection=true`) e lo chiude in `closeStrategy`. I campi SSH (`ssh`, `sshHost`, `sshPort`, `sshUser`, `sshPassword`, `sshKeyFile`, `sshPassphrase`) sono salvati in `connections.ini`; `sshPassword` e `sshPassphrase` sono **segreti** (`SECRET_FIELDS`) gestiti come la password DB: mai rimandati al browser, riusati se il form li lascia vuoti.

### Convenzione EJSON (il punto più delicato)

I documenti viaggiano come **Extended JSON**: il server serializza con `EJSON.serialize(..., { relaxed: true })` (numeri normali, ma ObjectId/Date restano `$oid`/`$date`) e fa il parse degli input utente con `EJSON.parse(...)` — `relaxed: false` per MongoDB (preserva i tipi BSON), `relaxed: true` per MySQL (produce tipi JS nativi da usare come parametri SQL). Vale anche per le righe MySQL: le colonne DATETIME arrivano al client come `$date`. Il frontend riconosce e renderizza le forme `$oid`, `$date`, `$numberLong`, ecc. (vedi `displayValue`/`simplify` in app.js) e l'editing inline sceglie il controllo in base al tipo (`valueType`/`buildEditor`). Gli `_id` vengono passati dal client come stringhe EJSON (`JSON.stringify(doc._id)`) e riconvertiti lato server dalla strategia (`parseId` per Mongo, con promozione automatica delle stringhe di 24 hex a ObjectId via `promoteObjectIds`; `parseRowId` per MySQL). Qualsiasi modifica al flusso dati deve preservare questa convenzione su entrambi i lati.
